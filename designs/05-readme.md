# Design: README.md

## Goal

Ship a top-level `README.md` that is enough to (1) convince a
drive-by visitor what `webtransportd` is and why they'd use it,
(2) get them a running daemon in 60 seconds, and (3) teach them
the one byte-level wire-format detail (the framing spec) they
need to write a non-trivial child. Docs must describe the real
shipping behaviour, not planned features — so this design lands
AFTER `--cert=auto` (cycle 42) is in place so the quickstart can
use it instead of telling new users to generate a PEM pair.

## Why now

- Cycle 42's `--cert=auto` removed the "first, generate a cert"
  prerequisite from the quickstart. With that, the shortest path
  from `git clone` to "I can talk to it" is three commands, and
  a README is the only thing needed to turn that into community
  adoption.
- The project already has `LICENSE`, `AUTHORS`, `CHANGES`, and
  `designs/`. A `README.md` is the one remaining "top-level file
  a reader expects."

## Scope — what ships in this cycle

One file: `README.md` at the repo root. No auto-generated
sections; no doc site; no man-page. Links out to the existing
`LICENSE`, `AUTHORS`, `CHANGES`, and `designs/` directories where
relevant.

Sections, in order (short headings for scannability):

1. **What it is.** One paragraph. Positions the project relative
   to [websocketd](https://github.com/joewalnes/websocketd) —
   "WebTransport/QUIC daemon that exec's a child per session and
   pipes bytes between the WT stream/datagram channel and the
   child's stdin/stdout." One sentence on why: any programming
   language that can read stdin and write stdout can serve a
   WebTransport session without linking a QUIC library.

2. **Quickstart.** Three shell blocks:
   ```bash
   git clone https://github.com/fire/webtransportd
   cd webtransportd
   make
   ./webtransportd --server --cert=auto --port=4433 \
       --exec=./examples/echo
   ```
   Followed by a ~10-line browser-side snippet that uses the
   native `WebTransport` JS API to connect to
   `https://localhost:4433/`, send a byte, and read the echo.
   Note that `--cert=auto` is a self-signed cert so the browser
   needs the `ignoreCertificateErrors` DevTools flag (or a manual
   trust-anchor import). Keep the snippet minimal — the point is
   "it works," not a full UI.

3. **Framing.** Reproduce (or link to) the `frame.h` header
   comment that documents the flag + varint + payload format.
   One short paragraph + the exact byte layout in a code block.
   This is the one detail a child author MUST get right, so it
   belongs in the README rather than under `designs/`.

4. **CLI flags.** A table, one row per flag, columns
   `Flag / Default / Meaning`. Generated by hand and kept in
   sync with `print_usage` in `webtransportd.c`. Cycle 42
   currently omits `--cert=auto` from `print_usage` — that doc
   update ships with this design too.

5. **Building from source.** One paragraph per platform:
   - Linux / macOS: `make` (uses system compiler, ASAN in the
     test build, no sanitizers in the release binary from CI).
     Note the `NO_SANITIZER=1` knob for producing a portable
     binary locally.
   - Windows: `MSYS2 + mingw-w64`. Call out the embedded
     UTF-8 manifest and the Godot-derived link recipe
     (`-lws2_32 -liphlpapi -lbcrypt
     -Wl,--allow-multiple-definition`). Cross-building from
     macOS/Linux to Windows via `x86_64-w64-mingw32-gcc` is
     supported and documented with a one-liner example.

6. **Relationship to Godot.** Brief: `thirdparty/picoquic`,
   `thirdparty/picotls`, `thirdparty/mbedtls`, and
   `thirdparty/picoquic/picoquic_mbedtls` are vendored
   **snapshots** pulled from Godot's tree — no SCsub, no
   `#include` from Godot core. The daemon is intentionally
   independent so an operator can run it without a Godot
   toolchain. Point readers at the Godot `modules/http3/`
   if they want to use the same stack inside the engine.

7. **License and contact.** One line pointing to `LICENSE`
   (BSD-2-Clause for first-party code) and `AUTHORS`. No
   changelog duplication — one line pointing at `CHANGES`.

Explicitly out of scope for cycle 44:
- No architecture diagrams. `designs/` is the design-doc home.
- No API reference (the daemon has no public C API — it's a
  binary + a child-process protocol).
- No benchmark numbers.
- No screenshots, no animated GIFs.

## Dependencies / blockers

- **Resolved**: `--cert=auto` (cycle 42) is live on `main` —
  the quickstart's `--cert=auto` line is valid.
- **Small prerequisite**: `print_usage` in `webtransportd.c`
  still advertises `--cert=<pem>`, not `--cert=<pem|auto>`. Fix
  that one line in the same commit as the README so the CLI
  flags table stays truthful.

## Verification

- `./webtransportd --help` output and the README's CLI flags
  table agree on every flag name, default, and meaning. A
  prospective reader who reads one and tries the other sees no
  drift.
- The quickstart shell block runs end-to-end on a fresh clone
  on macOS and Linux without any local prerequisite beyond
  `git`, `make`, and a C compiler.
- The browser snippet opens a WebTransport session against the
  running daemon and round-trips a byte. (Manual test with
  Chrome's DevTools — automated end-to-end is out of scope;
  `handshake_echo_test` covers the server-side round-trip.)
- No broken links inside `README.md` (relative links to
  `LICENSE`, `AUTHORS`, `CHANGES`, `designs/`, `examples/echo.c`,
  `examples/frame-helper.sh`).

## Commit shape

Single commit:

```
Cycle 44: top-level README.md

Ship the quickstart, framing spec, CLI table, platform build
notes, Godot-relationship note, and license/contact footer.
Also update print_usage to advertise --cert=auto so the README's

```

Files touched: `README.md` (new), `webtransportd.c`
(`print_usage` update).

## Risk notes

- **Browser snippet drift.** The native WebTransport JS API is
  still stabilising across browsers. The snippet is labelled
  "minimal, Chrome-tested" and kept under 15 lines. If it ever
  stops working, it's isolated enough to fix in a 1-line PR
  instead of a doc rewrite.
- **CLI table vs `--help` drift.** Fix this by making the
  verification step (`--help` output agrees with table) a
  convention: every `--<flag>` change to `webtransportd.c`
  requires an accompanying README table edit in the same
  commit. The `CHANGES` log will keep this obvious.
